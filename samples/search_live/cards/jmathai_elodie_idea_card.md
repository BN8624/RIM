## 한 줄 결론
메타데이터 기반의 파일 자동 정리 패턴과 --dry-run을 통한 안전한 실행 구조가 핵심이며, 업무 자동화 도구로 적용 가능하다.

## 왜 사람들이 관심 가졌나
방대한 양의 사진, 영상, 오디오 데이터를 EXIF 정보를 이용해 체계적으로 자동 분류하고 관리하여 수동 정리 시간을 줄이고 싶어 함.

## 실제 사용자 고통
- 특정 RAW 파일(Nikon Zf, Leica M11-D)이 유효한 미디어로 인식되지 않는 호환성 문제
- 파일명 정제 과정에서 닫는 괄호 ')'가 제거되는 문자열 처리 버그
- 동일 파일명 존재 시 덮어쓰기로 인한 데이터 손실 위험

## 기능 요청 신호
- Immich와 같은 외부 UI를 통한 양방향 앨범/즐겨찾기 관리 플러그인
- 모든 명령어에 --dry-run 플래그를 지원하여 변경 사항 사전 확인
- 중복 파일 이름 발생 시 덮어쓰지 않고 복사 번호를 추가하는 기능

## 워크플로우/자동화 신호
- 위치 정보가 없는 'Unknown Location' 파일들을 추출하여 수동으로 보완하고 다시 가져오는 효율적인 프로세스 부족

## 가져올 패턴
외부 메타데이터 추출 도구(ExifTool)를 래핑하여 파일 시스템 구조를 자동 정의하고, 실제 적용 전 시뮬레이션(--dry-run)을 제공하는 워크플로우.

## 버릴 것
- MapQuest API 연동 (ExifTool 내장 기능으로 대체 가능)
- 특정 클라우드 서비스(Dropbox, Google Photos)의 세부 API 구현체

## Dependency / Runtime Risk
- level: medium
- reason: OS 레벨의 외부 바이너리인 ExifTool 설치가 필수적이며, 실행 환경의 OS별 설치 과정 및 버전 호환성 관리가 필요함.

## 내 현재 병목에 적용
- area: 업무 자동화/OCR
- related_project: Repo Idea Miner
- reason: 파일의 메타데이터를 분석해 자동으로 분류/정리하는 로직은 다양한 문서 및 미디어 자동화 도구에 즉시 적용 가능한 패턴임.

## 1일 MVP
- status: 가능
- feature: EXIF 날짜/위치 기반 폴더 자동 분류기
- input: 미정리 사진/영상 폴더
- output: YYYY/MM/DD/도시명 구조로 정리된 파일 시스템
- excluded_scope: 클라우드 API 연동, 복잡한 RAW 파일 포맷의 정밀 지원, DB 기반의 상태 관리
- reason: ExifTool 래퍼 라이브러리를 사용하면 핵심 분류 및 이동 로직은 매우 단순하게 구현 가능함.

## 1일 Pattern PoC
- status: 가능
- idea: 파일 메타데이터 기반의 'Dry-run' 파일 이동 시뮬레이터
- input: 파일 경로 및 분류 규칙
- output: 실제 이동될 경로 리스트 (Log)
- reason: 파일 시스템 변경 전 변경 사항을 미리 보여주는 패턴은 모든 자동화 도구의 신뢰성을 높이는 필수 요소임.

## 만들면 망하는 이유
- 중복 파일 처리 미흡으로 인한 데이터 유실 발생 시 도구의 신뢰도가 급격히 하락함
- 특정 전문 포맷(RAW) 지원 누락으로 인해 핵심 타겟 유저인 사진작가들의 외면을 받을 수 있음

## 왜 이 판정인가
- 메타데이터 기반 자동 분류라는 명확하고 실용적인 유틸리티성
- 안전한 실행을 위한 dry-run 패턴의 완성도가 높음
- 단순 CLI를 넘어 외부 UI(Immich)와의 연동 니즈가 확인되어 확장 가능성이 있음

## 다음 행동
유사한 파일 정리 도구(예: PhotoPrism, Immich)의 메타데이터 처리 방식과 비교 분석
