## 한 줄 결론
파이썬 네이티브 DAG 기반의 증분 빌드 패턴은 분석 파이프라인의 효율성을 극대화할 수 있는 핵심 구조다.

## 왜 사람들이 관심 가졌나
YAML이나 별도의 DSL 학습 없이 순수 파이썬 함수만으로 복잡한 태스크 의존성을 정의하고, 변경된 부분만 선택적으로 실행하여 리소스를 절약하고 싶어함.

## 실제 사용자 고통
- 의존성 추적 오류로 인해 실제로는 변경되었음에도 up-to-date로 표시되어 빌드가 누락되는 현상
- Python 3.12+ 환경에서 멀티프로세싱 사용 시 fork() 관련 데드락 위험 및 경고 발생
- 동적으로 생성된 쉘 커맨드의 실제 내용을 확인하기 어려워 트러블슈팅에 제약이 있음

## 기능 요청 신호
- pyproject.toml 파일의 재귀적 검색 및 설정 로드 기능
- pre-commit 훅으로 사용하기 위해 파일 수정 시 non-zero exit code를 반환하는 옵션
- 기본 태스크로 'list' 명령어를 지정할 수 있는 설정

## 워크플로우/자동화 신호
- 파일 수정 여부에 따른 실패 반환 옵션이 없어 pre-commit 훅으로 통합하는 워크플로우가 불가능함

## 가져올 패턴
Python-native DAG Task Runner with Incremental Build (File-based dependency tracking)

## 버릴 것
- 복잡한 CLI 명령어 인터페이스 및 옵션 세트
- 범용성을 위한 과도한 플러그인 아키텍처
- 레거시 파이썬 버전 호환성 유지 로직

## Dependency / Runtime Risk
- level: low
- reason: 외부 서비스 의존성 없이 파이썬 표준 라이브러리를 기반으로 동작하는 로컬 CLI 도구임.

## 내 현재 병목에 적용
- area: 코딩 하네스/검증
- related_project: Repo Idea Miner
- reason: 레포 분석 파이프라인(Scout -> Judge)의 각 단계를 태스크로 정의하고, 입력 데이터나 분석 로직이 변경된 레포만 다시 분석하는 증분 실행 구조를 통해 분석 비용을 획기적으로 줄일 수 있음.

## 1일 MVP
- status: 가능
- feature: 파일 타임스탬프 기반의 단순 증분 태스크 실행기
- input: 태스크 정의 함수(의존 파일, 결과 파일 정의 포함)
- output: 변경된 파일이 있는 태스크만 순차적으로 실행한 결과
- excluded_scope: 병렬 실행(Multiprocessing), 플러그인 시스템, 복잡한 CLI 명령어
- reason: DAG 정렬 알고리즘과 파일 수정 시간 비교라는 핵심 로직만으로 1일 내 구현 가능함.

## 1일 Pattern PoC
- status: 가능
- idea: 분석 단계별 결과물을 파일로 저장하고, 입력 레포의 소스나 분석 프롬프트가 변경되었을 때만 해당 단계부터 재실행하는 파이프라인 구축
- input: 분석 대상 레포 리스트 및 분석 설정 파일
- output: 최종 합성된 분석 리포트
- reason: doit의 'targets'와 'file_dep' 개념을 차용하여 분석 파이프라인의 상태를 관리할 수 있음.

## 만들면 망하는 이유
- 의존성 추적 로직이 정교하지 못해 '업데이트 됨'으로 오판할 경우, 잘못된 분석 결과가 유지되는 데이터 오염 발생 가능
- 멀티프로세싱 구현 시 파이썬의 GIL 및 fork() 특성으로 인한 데드락이 발생하여 파이프라인이 멈출 위험

## 왜 이 판정인가
- 대규모 레포 분석 시 모든 단계를 매번 재실행하는 것은 불가능하며, 증분 빌드 패턴은 이를 해결할 유일한 방법임
- DSL 없이 파이썬 함수로 태스크를 정의하는 방식은 분석 로직의 유연한 변경과 빠른 실험을 가능하게 함
