# Repo Idea Card

레포: PrefectHQ/prefect

## 판정
DROP

## FAST DROP에 가까운가
NO

## 점수
3

## 한 줄 결론
워크플로우 오케스트레이션의 상태 관리 및 재시도 패턴은 유용하나, 인프라 의존성이 너무 높고 구현 복잡도가 커서 1일 MVP 수준으로 추출하기 어렵다.

## 왜 사람들이 관심 가졌나
파이썬 스크립트를 프로덕션 수준의 안정적인 데이터 파이프라인으로 쉽게 전환하고, 스케줄링과 모니터링을 통합 관리하기 위해 사용함.

## 실제 사용자 고통
- Redis 연결 누수 및 페일오버 시 서비스 중단 현상
- 백그라운드 서비스의 헬스체크 부재로 인한 장애 감지 지연
- UI 파라미터 입력의 불편함 및 기본값 덮어쓰기 버그

## 기능 요청 신호
- 지리 공간 BBOX 전용 UI 위젯 지원
- 백그라운드 서비스 전용 헬스체크 API
- Azure 관리 ID 인증 지원

## 워크플로우/자동화 신호
- 인프라 장애(Redis failover) 시 포드 상태는 정상이지만 실제 스케줄링은 중단되는 가시성 부족으로 인한 대응 지연

## 가져올 패턴
상태 기반 워크플로우 오케스트레이션 (State-based Workflow Orchestration)

## 버릴 것
- Prefect Cloud의 관리형 서비스 기능
- 복잡한 분산 서버-워커 아키텍처
- 다양한 클라우드 SDK 통합 레이어

## Dependency / Runtime Risk
- level: medium
- reason: PostgreSQL, Redis 등 외부 인프라 의존성이 강하며, 비동기 이벤트 루프 관리 및 상태 동기화의 복잡성이 높음

## 내 현재 병목에 적용
- area: 적용 부적합
- related_project: Repo Idea Miner
- reason: 특정 기능 패턴이라기보다 거대한 플랫폼 프레임워크에 가까워, 소규모 프로젝트에 부분적으로 이식하기에는 아키텍처 오버헤드가 너무 큼

## 1일 MVP
- status: 축소 불가
- feature: 간이 워크플로우 실행기
- input: 태스크 정의 및 의존성 그래프
- output: 태스크 실행 결과 및 상태 로그
- excluded_scope: 분산 서버/워커 구조, Redis/DB 기반 상태 저장, UI 모니터링 대시보드
- reason: 핵심 가치인 '회복 탄력성'과 '중앙 집중식 모니터링'을 제거하면 단순한 스크립트 실행기와 다를 바 없어 MVP로서의 가치가 없음

## 1일 Pattern PoC
- status: 가능
- idea: 데코레이터를 이용한 태스크 재시도 및 상태 기록 래퍼
- input: 실패 가능성이 있는 파이썬 함수
- output: 재시도 횟수와 최종 상태가 기록된 실행 결과
- reason: 오케스트레이션의 가장 작은 단위인 'Retry'와 'State' 기록 패턴은 단독으로 가볍게 구현 가능함

## 만들면 망하는 이유
- 인프라 설정(DB, Redis) 비용이 실제 기능 구현 비용보다 커질 가능성이 높음
- 단순 자동화 도구에 도입하기에는 아키텍처가 너무 무거워 배보다 배꼽이 더 큰 상황이 됨

## 왜 이 판정인가
- 구현 무게(Implementation Weight)가 Heavy하며, 1일 MVP로 구현 가능한 수준의 핵심 패턴이 모호함
- 특정 기능을 추출해 적용하는 '패턴'이 아니라 전체 시스템을 구축해야 하는 '플랫폼' 성격이 강함

## 다음 행동
더 가벼운 태스크 큐(예: Celery, RQ)나 단순 DAG 라이브러리와 비교하여 최소 기능 패턴 탐색
