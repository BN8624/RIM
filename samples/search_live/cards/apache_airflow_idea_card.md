# Repo Idea Card

레포: apache/airflow

## 판정
DROP

## FAST DROP에 가까운가
NO

## 점수
2

## 한 줄 결론
DAG 기반 워크플로우 오케스트레이션 패턴은 훌륭하나, 분산 아키텍처의 복잡성과 인프라 의존성이 너무 높아 1일 MVP 구현이 불가능하다.

## 왜 사람들이 관심 가졌나
복잡한 데이터 파이프라인을 코드로 정의하고 스케줄링하며 모니터링할 수 있는 표준화된 방법론을 제공하기 때문

## 실제 사용자 고통
- 입력 값 검증 부족으로 인한 모호한 에러 메시지 및 silent failure (#69595)
- 실행 중인 태스크 로그 테일링 시 발생하는 크래시
- TLS 프록시 환경에서의 보안 설정(Secure 플래그) 누락 문제

## 기능 요청 신호
- Airflow UI에서 Python 외의 언어(Java 등)로 작성된 태스크 구현 코드를 확인할 수 있는 기능 (AIP-108)

## 워크플로우/자동화 신호
- 버전 업그레이드(예: 3.1.8 -> 3.2.2) 시 발생하는 DB 마이그레이션 및 환경 업데이트의 높은 복잡도 (#68483)

## 가져올 패턴
DAG(Directed Acyclic Graph) 기반의 태스크 의존성 관리 및 상태 기반 실행 엔진

## 버릴 것
- 분산 스케줄러/워커 아키텍처
- Kubernetes 통합 레이어
- 복잡한 DB 마이그레이션 로직
- 방대한 외부 Provider 라이브러리

## Dependency / Runtime Risk
- level: high
- reason: PostgreSQL/MySQL 등 메타데이터 DB와 Kubernetes 등 외부 인프라에 대한 강한 의존성을 가지며, 분산 환경에서의 상태 동기화 리스크가 큼

## 내 현재 병목에 적용
- area: 업무 자동화/OCR
- related_project: Repo Idea Miner
- reason: 워크플로우 자동화 엔진의 핵심 패턴을 추출하여 단순화된 자동화 도구의 실행 순서 제어 로직에 적용 가능

## 1일 MVP
- status: 축소 불가
- feature: 단순 DAG 실행기
- input: 태스크 간 의존성이 정의된 설정 파일
- output: 의존성에 따른 순차적/병렬적 태스크 실행 결과
- excluded_scope: 분산 처리, 웹 UI, DB 영속성, 스케줄링
- reason: 핵심 가치인 '오케스트레이션'을 구현하려면 최소한의 상태 관리와 스케줄러가 필요한데, 이를 1일 안에 안정적으로 구현하기엔 범위가 너무 넓고 복잡함

## 1일 Pattern PoC
- status: 가능
- idea: Python 데코레이터를 이용한 단순 DAG 의존성 정의 및 로컬 실행기
- input: 함수 간 의존 관계 정의
- output: 의존성에 따른 함수 실행 순서 제어
- reason: 분산 환경과 인프라를 제외한 순수 Python 로직만으로는 DAG 실행 패턴의 PoC가 가능함

## 만들면 망하는 이유
- 인프라 설정(DB, K8s)에 너무 많은 시간이 소요되어 실제 비즈니스 로직 구현 전 포기할 가능성이 높음
- 분산 시스템의 동기화 및 상태 관리 복잡도로 인해 버그 수정에 대부분의 시간을 소비하게 됨

## 왜 이 판정인가
- 엔터프라이즈급 규모의 프로젝트로, 단순 패턴 추출 외에 전체적인 구조를 모방하여 MVP를 만들기에는 오버엔지니어링임
